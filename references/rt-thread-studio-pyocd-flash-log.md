# RT-Thread Studio PyOCD Flash Log Reference

Use this reference when analyzing firmware download or flashing logs from RT-Thread Studio that invoke RealThread PyOCD for a confirmed DAPLink, CMSIS-DAP, or DAP debug probe.

Do not choose PyOCD only because `pyocd.exe` exists on the PC. First confirm that the project or user is using DAPLink/CMSIS-DAP/DAP. If the user uses ST-LINK or J-Link, select the matching ST or SEGGER flashing tool instead.

## Typical Flash Flow

RT-Thread Studio may change into its bundled PyOCD directory and then run `pyocd.exe flash`.

Example command shape:

```powershell
cd /d D:\RT-ThreadStudio\repo\Extract\Debugger_Support_Packages\RealThread\PyOCD\0.1.3
pyocd.exe flash --target=STM32F407VG --erase=auto --frequency=1000000 D:\RT-Thread_project\Bluetooth_audio\Debug\rtthread.bin
```

Interpret this as:

- `pyocd.exe`: RT-Thread Studio bundled PyOCD flashing tool.
- `flash`: PyOCD flash operation.
- `--target=STM32F407VG`: target MCU name.
- `--erase=auto`: PyOCD chooses the erase strategy.
- `--frequency=1000000`: SWD/debug frequency, here 1 MHz.
- `D:\...\rtthread.bin`: firmware image generated by the build step.

This is the flashing stage, not the compilation stage. Compilation is typically driven by `make` and the ARM GCC toolchain, while flashing is driven by `pyocd.exe flash`.

This PyOCD path is appropriate for DAPLink/CMSIS-DAP/DAP workflows. It is not a generic default for every STM32 board.

## Success Pattern

A successful flash log may include a progress bar and a loader summary like:

```text
Erased 262144 bytes (6 sectors), programmed 262144 bytes (256 pages), skipped 0 bytes (0 pages) at 23.99 kB/s
```

Treat this as evidence that flash erase and programming completed.

Record:

- target MCU
- firmware path
- erase mode
- frequency
- erased bytes
- programmed bytes
- flashing speed

## Unexpected ACK / Transfer Error

If the log contains:

```text
Unexpected ACK value
TransferError
```

or similar CMSIS-DAP / DAPLink communication errors, stop treating the issue as a software-only problem.

Immediately ask the user to check hardware/debug connection details:

- SWDIO wiring
- SWCLK wiring
- GND shared between board and debugger
- target board power and 3.3 V supply
- reset line or reset button state
- BOOT pin state
- debugger USB connection
- whether the selected target MCU is correct
- whether another tool is occupying the debugger
- whether lowering `--frequency` helps, for example from `1000000` to `500000` or lower

Do not make code changes merely to fix an `Unexpected ACK` flash failure unless later evidence points to firmware locking or reset behavior caused by the application.

## PyOCD Runtime Extraction Error

If the log contains:

```text
VCRUNTIME140.dll could not be extracted!
fopen: Permission denied
```

or access errors involving a temporary `_MEI...` directory, PyOCD failed before it reached the target MCU.

Treat this as a PC-side PyOCD runtime extraction or permission problem, not as a firmware, build, or board-wiring problem.

When this happens inside an agent sandbox or restricted shell, retry the exact RT-Thread Studio command from a normal user shell or with approved outside-sandbox execution before blaming the user's hardware setup. The same command may succeed outside the sandbox.

Recommended checks:

- close other RT-Thread Studio or `pyocd.exe` processes that may hold the temporary extraction directory
- check whether antivirus or Windows Controlled Folder Access is blocking the bundled PyOCD executable
- check whether the user can write to `%TEMP%`
- try setting `TEMP` and `TMP` to a writable directory before running PyOCD
- if the command is being run by an agent sandbox, request outside-sandbox execution or ask the user to run the same command in a normal Command Prompt
- check whether the PyOCD directory is writable and not read-only
- retry RT-Thread Studio download after cleaning stale `_MEI...` temporary folders when they are not in use

## Analysis Rules

- Flash only after a successful build and after the firmware path is known.
- Confirm the programmer/debugger type before selecting this PyOCD flow.
- Use this flow when the confirmed programmer/debugger type is DAPLink, CMSIS-DAP, or DAP.
- Do not use this flow for ST-LINK or J-Link unless the project explicitly documents that those probes are routed through PyOCD.
- Record the exact `pyocd.exe flash` command.
- Distinguish flash failures from build failures.
- Treat `Unexpected ACK value` as probe-target communication failure evidence.
- Ask for wiring, power, reset, BOOT, debugger, and target checks before continuing with software changes.
- Treat `VCRUNTIME140.dll could not be extracted` as a PyOCD runtime extraction or permission problem before hardware communication.
- If `pyocd.exe flash` succeeds from a normal shell but fails in the agent shell, report it as an agent/sandbox execution limitation.
- Retry flashing only when there is a concrete change in hardware state, debugger state, target selection, frequency, or user confirmation.
